# v1.3.0 — `CanSave(target)` Overload + Auth Method Triplication Fix

**Release Date:** 2026-04-13
**NuGet:** [Neatoo.RemoteFactory 1.3.0](https://nuget.org/packages/Neatoo.RemoteFactory/1.3.0)
**Breaking changes:** Additive interface change to `IFactorySave<T>` — see [Migration Guide](#migration-guide) below.

---

## Overview

Two improvements to authorization handling on save operations:

1. **`CanSave(target)` overload.** When an authorization class declares a `Write`-scoped method that takes a target entity parameter (e.g., `bool CanWrite(IOrder target)`), the generator now produces a `CanSave(T target, CancellationToken)` overload on `IFactorySave<T>` that runs both non-target role checks AND target-aware entity-state checks. Previously the generator suppressed `CanSave` generation entirely when target-parameterized auth was present, which was inconsistent with how other parameterized auth methods (e.g., `CanFetch(Guid orderId)`) work.
2. **Auth method triplication fix.** Generated `LocalCanSave` / `LocalSave` previously called each authorization method three times (once per Insert/Update/Delete contribution) due to a record-equality bug in `AuthMethodCall` and `AspAuthorizeCall`. `Distinct()` was using reference equality on the embedded parameter lists. Generated output now emits each auth method exactly once.

No application code changes are required for the common case (callers only saw cosmetic noise in generated code; the overload addition is opt-in).

---

## What's New

### `CanSave(T target, CancellationToken)` overload on `IFactorySave<T>`

```csharp
// Before (v1.2.0): only the parameterless overload existed
Task<Authorized> CanSave(CancellationToken cancellationToken = default);

// After (v1.3.0): both overloads available
Task<Authorized> CanSave(CancellationToken cancellationToken = default);
Task<Authorized> CanSave(T target, CancellationToken cancellationToken = default);
```

Behavior:
- **`CanSave()`** runs only non-target authorization methods (role checks, permission flags). Unchanged.
- **`CanSave(target)`** runs ALL matching `Write`-scoped auth methods — both non-target AND target-parameterized. Same dispatch pattern as `CanFetch(key)` calling multiple matching methods.

When to call which:
- **`CanSave()`** for UI gating before the user has filled in the entity (e.g., showing/hiding a "Save" button based on role).
- **`CanSave(target)`** when the entity is in hand and you want target-aware auth (e.g., "this user can edit orders, BUT not orders in `Shipped` state").

The generator routes to whichever auth methods match. If your auth class has only role checks, `CanSave(target)` returns the same result as `CanSave()`. If your auth class has only target-parameterized methods, `CanSave()` short-circuits to `Authorized(true)` (no non-target methods to run).

## Bug Fixes

### Auth method triplication in generated `LocalCanSave` / `LocalSave`

`BuildSaveMethodFromGroup` in `FactoryModelBuilder.cs` merges authorization from Insert, Update, and Delete methods using `SelectMany().Distinct()`. `AuthMethodCall` is a `record` with `IReadOnlyList<ParameterModel>` properties; the compiler-synthesized `Equals` used reference equality on those lists, so identical auth method calls contributed by Insert/Update/Delete (with separate parameter list instances) were treated as distinct, and `Distinct()` failed to deduplicate. The generated `LocalCanSave()` then emitted three calls to the same auth method.

Fix: implemented value-based `Equals`/`GetHashCode` on `AuthMethodCall` and `AspAuthorizeCall` — comparing parameter lists element-wise via `SequenceEqual`. Generated `LocalCanSave` and `LocalSave` now emit each auth method exactly once.

Behavior impact: none (auth methods are idempotent), but generated code is materially smaller and no longer looks broken to users inspecting it. Two new regression tests (`AuthMethodCallEqualityTests`, `AuthMethodTriplicationTests`) guard the fix.

## Breaking Changes

### `IFactorySave<T>` gains `CanSave(T target, CancellationToken)`

The interface now requires implementors to provide the new overload.

**Who is affected:** projects that manually implement `IFactorySave<T>` (instead of relying on the generator). In practice this is rare — the generator emits `IFactorySave<T>` implementations for any `[Factory]` class with Save semantics.

**Who is NOT affected:** any project that lets the generator produce its `IFactorySave<T>` implementations. The generator updates automatically; no source changes needed.

## Migration Guide

### If you manually implement `IFactorySave<T>`

Add the new overload alongside the existing `CanSave(CancellationToken)`:

```csharp
public Task<Authorized> CanSave(MyEntity target, CancellationToken cancellationToken = default)
{
    // For implementations without target-aware auth, just delegate:
    return CanSave(cancellationToken);
}
```

If you have target-aware authorization logic, implement it here (typically by checking entity state alongside the role/permission checks already in `CanSave()`).

### If you use the generator (the common case)

No action required. Rebuild and the generator emits both overloads automatically. To take advantage of `CanSave(target)`, declare an auth method that takes a target type parameter:

```csharp
public class OrderAuth
{
    [AuthorizeFactory(AuthorizeFactoryOperation.Write)]
    public bool CanWriteRole(IUserContext user) => user.HasRole("OrderManager");

    [AuthorizeFactory(AuthorizeFactoryOperation.Write)]
    public bool CanWrite(IOrder target) => target.Status != OrderStatus.Shipped;
}
```

Now `myOrderFactory.CanSave(order)` runs both `CanWriteRole(user)` AND `CanWrite(order)`, returning `false` if either rejects.
