# v0.21.0 - Can* Method Guard Derived from Auth Class

**Release Date:** 2026-03-08
**NuGet:** [Neatoo.RemoteFactory 0.21.0](https://nuget.org/packages/Neatoo.RemoteFactory/0.21.0)

## Overview

Can* method guard and remote behavior now derives from the auth class methods instead of the parent factory method. This fixes a bug where `CanCreate()`, `CanFetch()`, etc. threw "Server-only method called in non-server runtime" on the client when the factory method was `[Remote] internal` but the auth class had `public` methods with no server-only dependencies.

## What's New

- **Auth-method-driven Can* behavior.** Can* methods now follow the same accessibility paradigm as all other methods in RemoteFactory:
  - `public` auth methods: Can* runs on client, no guard, synchronous
  - `internal` auth methods (no `[Remote]`): Can* is server-only with `IsServerRuntime` guard
  - `[Remote] internal` auth methods: Can* routes to server via remote delegate, asynchronous
- **CanSave aggregation rule.** When CanSave aggregates auth methods from Insert, Update, and Delete, if ANY constituent auth method is `internal` or `[Remote]`, CanSave gets the guard (most restrictive wins for security).

## Breaking Changes

- **Can* methods may now run on the client instead of the server.** Previously, Can* methods inherited the `IsServerRuntime` guard from the parent factory method. If your factory method was `[Remote] internal`, the Can* method was server-only. Now, if your auth class has `public` methods, Can* runs on the client regardless of the factory method's visibility.

  **If you need server-only authorization checks**, add `[Remote] internal` to your auth interface methods:

  ```csharp
  // Before: Can* was server-only because factory method was [Remote] internal
  // After: Can* runs on client because auth methods are public
  public interface IMyAuth
  {
      [AuthorizeFactory(AuthorizeFactoryOperation.Create)]
      bool CanCreate();  // public => runs on client
  }

  // To keep Can* server-only, add [Remote] internal to auth methods:
  public interface IMyAuth
  {
      [Remote]
      [AuthorizeFactory(AuthorizeFactoryOperation.Create)]
      internal bool CanCreate();  // [Remote] internal => routes to server
  }
  ```

## Bug Fixes

- **Fixed Can* methods throwing on Blazor WASM client.** When a factory method was `[Remote] internal` but the auth class had `public` methods with client-available services, calling `CanCreate()` etc. on the client threw "Server-only method called in non-server runtime." This blocked client-side UI authorization checks (e.g., showing/hiding buttons based on user role).

## Migration Guide

### If your auth class has public methods (common case)

No changes needed. Can* methods now correctly run on the client. If they were previously throwing on the client, they will now work.

### If you relied on Can* being server-only

If your authorization logic depends on server-only services and you were relying on the factory method's `[Remote] internal` to make Can* server-only, you must now explicitly mark your auth interface methods as `[Remote] internal`:

```csharp
public interface IMyAuth
{
    [Remote]
    [AuthorizeFactory(AuthorizeFactoryOperation.Create)]
    internal bool CanCreate();
}
```
