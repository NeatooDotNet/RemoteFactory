# v0.21.0 - [Remote] Requires Internal, Trimming Fixes

**Release Date:** 2026-03-08
**NuGet:** [Neatoo.RemoteFactory 0.21.0](https://nuget.org/packages/Neatoo.RemoteFactory/0.21.0)

## Overview

`[Remote]` now requires `internal` visibility (NF0105 flipped), Can* method guards derive from auth class methods instead of factory methods, and transport serialization is trimming-safe with source-generated JSON contexts.

## What's New

- **`[Remote]` requires `internal`.** `[Remote] public` is now a compile-time error (NF0105). Use `[Remote] internal` — the generator promotes the method to `public` on the factory interface. This enables IL trimming of method bodies on client assemblies.
- **Auth-method-driven Can* behavior.** Can* methods now derive guard and remote behavior from the auth class methods, not the parent factory method:
  - `public` auth methods: Can* runs on client, no guard, synchronous
  - `internal` auth methods (no `[Remote]`): Can* is server-only with `IsServerRuntime` guard
  - `[Remote] internal` auth methods: Can* routes to server via remote delegate, asynchronous
- **`[AuthorizeFactory<T>]` Design project coverage.** New `AuthorizedOrder` and `AuthorizedOrderAuth` examples in the Design project with 12 tests demonstrating authorization patterns.
- **Trimming-safe transport serialization.** Source-generated `NeatooTransportJsonContext` for `RemoteRequestDto`, `RemoteResponseDto`, and `ObjectJson` eliminates runtime reflection during client-server communication in trimmed builds.
- **Trimming documentation.** Complete csproj configuration guide covering `PrivateAssets="all"`, `TrimmableAssembly`, and full project examples.

## Breaking Changes

- **`[Remote] public` is now a compile-time error (NF0105).** Change `[Remote] public` methods to `[Remote] internal`. The factory interface already exposes them as `public`.
- **Can* methods may now run on the client instead of the server.** Previously, Can* methods inherited the `IsServerRuntime` guard from the parent factory method. Now they derive from the auth class methods. If your auth class has `public` methods, Can* runs on the client regardless of the factory method's visibility. To keep Can* server-only, add `[Remote] internal` to your auth interface methods.
- **FactoryMode.RemoteOnly removed.** The compile-time concept is no longer needed since `[Remote] internal` handles the same intent through visibility.

## Bug Fixes

- **Fixed Can* methods throwing on Blazor WASM client.** When a factory method was `[Remote] internal` but the auth class had `public` methods, calling `CanCreate()` etc. on the client threw "Server-only method called in non-server runtime."
- **Fixed `Authorized<T>` deserialization failure in trimmed builds.** IL trimmer stripped constructor parameter metadata from `Authorized<T>`, causing "DeserializeNoConstructor" errors. Fixed with `[JsonConstructor]`, `[DynamicallyAccessedMembers]`, and `ILLink.Descriptors.xml`.
- **Fixed `ObjectJson` constructor parameter name trimming.** Added `[JsonConstructor]` attribute and source-generated JSON context for trimming-safe serialization.
- **Fixed camelCase mismatch in trimmed builds.** Source-generated JSON context now uses `JsonSerializerDefaults.Web` to match ASP.NET Core server serialization conventions.

## Migration Guide

### [Remote] public to [Remote] internal

```csharp
// Before (now compile error NF0105):
[Remote]
public void Fetch([Service] IRepository repo) { }

// After:
[Remote]
internal void Fetch([Service] IRepository repo) { }
```

The factory interface already exposes the method as `public` — no consumer-visible change.

### Can* method behavior

If your auth class has `public` methods and you relied on the factory method's `[Remote] internal` to make Can* server-only, explicitly mark auth interface methods:

```csharp
public interface IMyAuth
{
    [Remote]
    [AuthorizeFactory(AuthorizeFactoryOperation.Create)]
    internal bool CanCreate();  // [Remote] internal => routes to server
}
```
