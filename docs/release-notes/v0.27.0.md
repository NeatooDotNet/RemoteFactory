# v0.27.0 - Recursive Nested DTO Discovery for IL Trimming

**Release Date:** 2026-03-31
**NuGet:** [Neatoo.RemoteFactory 0.27.0](https://nuget.org/packages/Neatoo.RemoteFactory/0.27.0)

## Overview

The source generator now recursively discovers nested DTOs through property walking, so DTO types used as properties of other DTOs are automatically registered in `DtoConstructorRegistry` and survive IL trimming in Blazor WASM.

## What's New

- **Recursive nested DTO discovery** -- The generator walks public instance properties of discovered DTO types to find nested DTOs that also need constructor registration. Previously, only direct return types from factory method signatures were discovered. Now, if `ParentDto` has a `List<ChildDto>` property, both `ParentDto` and `ChildDto` are registered automatically.
- **Inherited property traversal** -- Properties inherited from base classes are also walked, so `DerivedDto : BaseDto` discovers DTOs from both levels.
- **Collection and nullable unwrapping** -- Property types wrapped in `List<T>`, `IReadOnlyList<T>`, arrays, and nullable annotations are unwrapped to find the inner DTO type.
- **Cycle detection** -- Circular references (A references B, B references A) and self-references are handled safely without infinite recursion.

## Breaking Changes

None. This is purely additive -- types that were not discovered before are now discovered. Over-discovery (registering types that are never deserialized) is harmless.

## Bug Fixes

- **Fix nested DTO deserialization failure under IL trimming** -- Discovered in zTreatment production: `AdminClinicAssignment` (a property of `AdminUserListItem`) was not registered in `DtoConstructorRegistry` because the generator only inspected direct return types. Users no longer need to manually register nested DTOs or return them from a factory method as a workaround.
