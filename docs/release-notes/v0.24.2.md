# v0.24.2 - Fix Event DI Validation on Blazor WASM

**Release Date:** 2026-03-27
**NuGet:** [Neatoo.RemoteFactory 0.24.2](https://nuget.org/packages/Neatoo.RemoteFactory/0.24.2)

## Overview

Fixes DI validation failures (`AggregateException`) when Blazor WASM clients reference domain assemblies containing `[Event]` factory methods. Two bugs caused the `AddNeatooRemoteFactory(NeatooFactory.Remote, ...)` call to register server-only event infrastructure that the client container could not construct.

## Bug Fixes

- **Fixed `IEventTracker` registered unconditionally across all factory modes.** `EventTracker` requires `ILogger<EventTracker>` in its constructor. On Blazor WASM clients with `ValidateOnBuild=true` and without logging configured, the DI container walked the dependency graph, failed to construct `EventTracker`, and threw `AggregateException`. The error cascaded to report unrelated services as unresolvable. Fixed by registering `NullEventTracker` (a no-op implementation with zero dependencies) for Remote mode, while Server and Logical modes continue to use the full `EventTracker`.

- **Fixed missing `IsServerRuntime` guard on class factory event registrations.** `ClassFactoryRenderer.RenderLocalEventRegistration` generated event delegate registrations that resolved `IEventTracker`, `IHostApplicationLifetime`, and `IServiceScopeFactory` without wrapping them in `if (NeatooRuntime.IsServerRuntime)`. The equivalent method in `StaticFactoryRenderer` already had this guard. Without the guard, the IL trimmer could not eliminate dead server-side event code from client assemblies. Fixed by adding the matching guard.

## Technical Details

The fix applies belt-and-suspenders: conditional `IEventTracker` registration (prevents DI validation failure) AND `IsServerRuntime` guard on class factory event registration (enables IL trimmer to eliminate server-only code).

**NullEventTracker** is a new `internal sealed` class that implements `IEventTracker` with no-op behavior: `Track()` does nothing, `WaitAllAsync()` returns `Task.CompletedTask`, and `PendingCount` returns 0. This preserves the public API contract -- `IEventTracker` is always resolvable regardless of factory mode -- while eliminating the logging dependency on Remote-mode clients.

**Registration logic** in `AddRemoteFactoryServices`:
- `NeatooFactory.Remote` registers `NullEventTracker`
- `NeatooFactory.Server` or `NeatooFactory.Logical` registers `EventTracker`
