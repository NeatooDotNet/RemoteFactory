# v0.23.2 - Fix DTO Deserialization Under IL Trimming

**Release Date:** 2026-03-25
**NuGet:** [Neatoo.RemoteFactory 0.23.2](https://nuget.org/packages/Neatoo.RemoteFactory/0.23.2)

## Overview

Fixes `DeserializeNoConstructor` exception when deserializing plain DTO classes in trimmed Blazor WASM applications. The IL trimmer strips constructor metadata that `System.Text.Json`'s `DefaultJsonTypeInfoResolver` discovers via reflection, breaking deserialization for non-Neatoo return types.

## Bug Fixes

- **Fixed DTO deserialization under IL trimming.** `NeatooJsonTypeInfoResolver.GetTypeInfo()` now provides an `Activator.CreateInstance` fallback for types not registered in DI. Previously, plain DTO classes (e.g., a `ProtocolComparisonData` returned by an Interface Factory) failed with `DeserializeNoConstructor` when the Blazor WASM app was published with trimming enabled. ([#48](https://github.com/NeatooDotNet/RemoteFactory/issues/48))

## Technical Details

The existing `NeatooJsonTypeInfoResolver` already set `CreateObject` for Neatoo types via DI when STJ's reflection-based discovery failed. The fix adds the missing `else if` branch for non-DI types with a public parameterless constructor, using `Activator.CreateInstance(type)`.

The `else if` guard (rather than a plain `else`) is necessary because `RecordBypassConverter.Read()` re-enters the resolver for record types. Without the constructor check, `Activator.CreateInstance` would throw `MissingMethodException` for records with only parameterized constructors.

## Workaround Removal

If you previously added a `TrimmerRootDescriptor` XML to preserve DTO types manually, you can remove it after upgrading:

```xml
<!-- No longer needed with v0.23.2 -->
<linker>
  <assembly fullname="MyApp.DomainModels">
    <type fullname="MyApp.DomainModels.MyDto" preserve="all" />
  </assembly>
</linker>
```
