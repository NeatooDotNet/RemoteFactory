# v1.5.0 — Delete `[Event]` Method Attribute API

**Release Date:** 2026-04-14
**NuGet:** [Neatoo.RemoteFactory 1.5.0](https://nuget.org/packages/Neatoo.RemoteFactory/1.5.0)
**Breaking changes:** Yes — the entire `[Event]` method attribute API and its supporting infrastructure are removed. Consumers using `[Event]` for fire-and-forget work must migrate to manual `Task.Run` + `IServiceScopeFactory.CreateScope()`. See [Migration Guide](#migration-guide) below.

> **Pre-stable API note.** v1.5.0 is an intentional minor bump carrying a breaking change. The 1.x line is still settling the event-shaped surface; breaking changes remain permitted under minor bumps until the surface stabilizes.

---

## Overview

`[Event]` and `[FactoryEventHandler<T>]` were two completely different features that, for most observers — including us — looked interchangeable. `[Event]` generated a named delegate for fire-and-forget work in an isolated DI scope, tracked by `IEventTracker` for graceful shutdown. `[FactoryEventHandler<T>]` dispatches transactional domain events that share the caller's scope and participate in the caller's DB transaction. Every review of the surface — and the repeatedly-CI-flaky `EventTrackerTests` that motivated a hard look at the feature — produced the same conclusion: two event-named pipelines with opposite execution semantics is a footgun. v1.5.0 deletes `[Event]`, keeps `[FactoryEventHandler<T>]`, and narrows the library to a single event-shaped abstraction.

Consumers who used `[Event]` for fire-and-forget work — send an email, publish to a queue, call a webhook — migrate to a two-line pattern: `Task.Run` inside the factory method, with a fresh DI scope created via `IServiceScopeFactory.CreateScope()`. RemoteFactory no longer owns that abstraction. See [Migration Guide](#migration-guide) below for the three cases the old `[Event]` feature handled implicitly: graceful shutdown, correlation propagation, and tenant / ambient context.

---

## Breaking Changes

### Deleted attribute and enum values

- `[Event]` method attribute (`EventAttribute` class)
- `FactoryOperation.Event`
- `AuthorizeFactoryOperation.Event = 512`

Consumer code with `[Event]` on a method will no longer compile — the C# compiler emits **CS0246** (`type or namespace 'Event' could not be found`). No RemoteFactory-prefixed diagnostic is raised because the generator no longer scans for `[Event]`.

Any `[AuthorizeFactory(AuthorizeFactoryOperation.Event)]` or `[AuthorizeFactory(AuthorizeFactoryOperation.Event | ...)]` references are now compile errors — the enum value is gone.

### Deleted types — fire-and-forget runtime

- `IEventTracker` / `EventTracker`
- `EventTrackerHostedService`
- `IEventScopeInitializer`
- `DelegateEventScopeInitializer`
- `CorrelationContextScopeInitializer`

### Deleted extension method

- `AddRemoteFactoryEventScopeInitializer(...)` (removed from the public service-collection surface)

### Deleted diagnostics

- **NF0401** — `[Event]` method must have `CancellationToken` as final parameter
- **NF0402** — `[Event]` method in a non-`[Factory]` class
- **NF0403** — `[Event]` method cannot be combined with `[Create]`/`[Fetch]`/etc.
- **NF0404** — `[Event]` method return type must be `Task`

### Deleted log events

Event IDs **9001-9009** (the `EventTracker` / event-scope category) are removed from `Neatoo.RemoteFactory` logging. Log filters that target those IDs can be deleted.

### Silent generator change — mixed `[Event]` + `[Create]` classes

A class that previously mixed `[Event]` methods with `[Create]`/`[Fetch]`/etc. methods still compiles in v1.5.0, with a behavior change: the `[Event]` attribute no longer exists, so the formerly-`[Event]` methods become plain `internal` methods with no generated delegate. The factory's non-event methods generate identically. Ensure you delete the old `[Event]` attribute usage during migration — a bare method left behind is no longer invocable as an event delegate.

## What's Still Here

The **factory-event** surface — a different feature from the deleted `[Event]` attribute — is **unchanged**:

| Feature | Status |
|---------|--------|
| `[FactoryEventHandler<T>]` class attribute | Unchanged (v1.4.0 surface) |
| `IFactoryEvents.Raise<T>(...)` inside factory methods | Unchanged |
| `RaiseOptions.ServerOnly` | Unchanged |
| Server-side static-method handler dispatch (shared scope, sequential, awaited) | Unchanged |
| `IFactoryEventRelay.Relay(IReadOnlyList<FactoryEventBase>)` client-side relay | Unchanged |
| `NoOpFactoryEventRelay` default in `NeatooFactory.Remote` mode | Unchanged |
| `FactoryEventBase` + `[FactoryEvent]` + trimming preservation | Unchanged |
| `ICorrelationContext` scoped DI registration | Unchanged — `MakeRemoteDelegateRequest` uses it for relay correlation IDs |
| Diagnostic **NF0405** (`[FactoryEventHandler<T>]` on wrong target) | Unchanged |
| Diagnostics NF0501-NF0503 (handler-method matching) | Unchanged |
| Log events 3008, 3009, 3011, 3012 (relay path) | Unchanged |

See [Factory Events](../factory-events.md) for the complete factory-event surface. **This is the one event-shaped feature RemoteFactory still ships.**

## Migration Guide

The `[Event]` attribute encapsulated three things, each of which is now the consumer's responsibility: a fresh DI scope, graceful-shutdown task tracking, and any ambient-context propagation (correlation ID, tenant, etc.). The migration pattern is a manual `Task.Run` inside the factory method, with a child scope created from `IServiceScopeFactory` and the ambient context copied in explicitly.

### Before

```csharp
[Factory]
internal static partial class EmailEvents
{
    [Remote, Event]
    internal static async Task _OnOrderPlaced(
        int orderId,
        decimal total,
        [Service] ISmtpClient smtp,
        CancellationToken ct)
    {
        await smtp.SendAsync("ops@example.com", "Order placed", $"{orderId} {total:C}", ct);
    }
}

// Fire-and-forget at the call site:
_ = EmailEvents.OnOrderPlacedEvent(orderId, total);
```

RemoteFactory's old generator built the `OnOrderPlacedEvent` delegate, created a fresh DI scope on every invocation, copied correlation / tenant context into it via `IEventScopeInitializer`, and tracked the outstanding `Task` in `IEventTracker` so `EventTrackerHostedService` could drain it during application shutdown.

### After

```csharp
// In your factory method (or any server-side code), raise fire-and-forget work
// with an explicit Task.Run, a fresh DI scope, and explicit context copy.
[Factory]
internal partial class Order
{
    [Remote, Create]
    internal async Task Create(
        int orderId,
        decimal total,
        [Service] AppDbContext db,
        [Service] IServiceScopeFactory scopeFactory,
        [Service] ICorrelationContext correlation,
        [Service] IHostApplicationLifetime lifetime,
        [Service] IFireAndForgetTracker tracker,   // consumer-owned, see Graceful Shutdown below
        CancellationToken ct)
    {
        db.Orders.Add(new OrderEntity(orderId, total));
        await db.SaveChangesAsync(ct);

        // Snapshot ambient context BEFORE entering Task.Run — the child scope
        // does not inherit it automatically.
        var correlationId = correlation.CorrelationId;
        var stopping = lifetime.ApplicationStopping;

        var task = Task.Run(async () =>
        {
            await using var scope = scopeFactory.CreateAsyncScope();

            // Re-assign the ambient context inside the child scope.
            var childCorrelation = scope.ServiceProvider.GetRequiredService<ICorrelationContext>();
            childCorrelation.CorrelationId = correlationId;

            var smtp = scope.ServiceProvider.GetRequiredService<ISmtpClient>();
            await smtp.SendAsync(
                "ops@example.com",
                "Order placed",
                $"{orderId} {total:C}",
                stopping);
        }, stopping);

        tracker.Track(task); // see Graceful Shutdown below
    }
}
```

Three things to get right, each covered below.

---

### Edge case 1 — Graceful shutdown

`[Event]`'s `IEventTracker` + `EventTrackerHostedService` waited for outstanding fire-and-forget work during `ApplicationStopping` so in-flight emails / webhooks / queue publishes completed before the process exited. RemoteFactory no longer owns this.

**Pattern:** a small consumer-owned tracker service, registered as a singleton, plus an `IHostedService` that awaits its pending tasks on `ApplicationStopping`:

```csharp
public interface IFireAndForgetTracker
{
    void Track(Task task);
    Task WaitAllAsync(CancellationToken ct);
}

public sealed class FireAndForgetTracker : IFireAndForgetTracker
{
    private readonly ConcurrentDictionary<Task, byte> _pending = new();

    public void Track(Task task)
    {
        _pending[task] = 0;
        task.ContinueWith(
            t => _pending.TryRemove(t, out _),
            TaskContinuationOptions.ExecuteSynchronously);
    }

    public Task WaitAllAsync(CancellationToken ct) =>
        Task.WhenAll(_pending.Keys).WaitAsync(ct);
}

public sealed class FireAndForgetShutdownHost : IHostedService
{
    private readonly IFireAndForgetTracker _tracker;
    private readonly IHostApplicationLifetime _lifetime;

    public FireAndForgetShutdownHost(
        IFireAndForgetTracker tracker,
        IHostApplicationLifetime lifetime)
    {
        _tracker = tracker;
        _lifetime = lifetime;
    }

    public Task StartAsync(CancellationToken ct) => Task.CompletedTask;

    public async Task StopAsync(CancellationToken ct)
    {
        try { await _tracker.WaitAllAsync(ct); }
        catch (OperationCanceledException) { /* shutdown budget exhausted */ }
    }
}

// DI:
services.AddSingleton<IFireAndForgetTracker, FireAndForgetTracker>();
services.AddHostedService<FireAndForgetShutdownHost>();
```

Call `tracker.Track(task)` on every `Task.Run` the factory method spawns. The hosted service awaits them on `ApplicationStopping` up to the host's shutdown timeout. Tests can call `tracker.WaitAllAsync(ct)` directly, mirroring the old `IEventTracker.WaitAllAsync` pattern.

---

### Edge case 2 — Correlation ID propagation

`CorrelationContextScopeInitializer` copied the parent scope's `ICorrelationContext.CorrelationId` into the child scope automatically. `ICorrelationContext` is **still registered scoped** — it's used by `MakeRemoteDelegateRequest` for relay correlation — but the automatic copy is gone.

**Pattern:** snapshot the value from the parent scope before `Task.Run`, re-assign it inside the child scope before resolving any correlation-aware services:

```csharp
// Parent scope (factory method):
var correlationId = correlation.CorrelationId; // ICorrelationContext injected as [Service]

var task = Task.Run(async () =>
{
    await using var scope = scopeFactory.CreateAsyncScope();

    // Child scope — re-assign BEFORE resolving anything that reads CorrelationId.
    var childCorrelation = scope.ServiceProvider.GetRequiredService<ICorrelationContext>();
    childCorrelation.CorrelationId = correlationId;

    // Now resolve services that log with the correlation ID.
    var logger = scope.ServiceProvider.GetRequiredService<ILogger<Whatever>>();
    // ...
}, stopping);
```

If you only use `ICorrelationContext` for logging and you use `ILogger`'s scope mechanism (`logger.BeginScope(...)`), you can wrap the `Task.Run` body in `logger.BeginScope` with the captured ID instead of mutating `ICorrelationContext`.

---

### Edge case 3 — Tenant / ambient context

If your `IEventScopeInitializer` implementation copied tenant ID, user ID, culture, or any other ambient scoped context into the child scope, the same snapshot-and-copy pattern applies. Capture in the parent, re-assign in the child.

```csharp
// Parent (factory method):
var tenantId = tenantContext.TenantId;  // [Service] ITenantContext
var userId   = userContext.UserId;       // [Service] IUserContext

var task = Task.Run(async () =>
{
    await using var scope = scopeFactory.CreateAsyncScope();

    scope.ServiceProvider.GetRequiredService<ITenantContext>().TenantId = tenantId;
    scope.ServiceProvider.GetRequiredService<IUserContext>().UserId = userId;

    // Child-scope services now see the same ambient context.
    var repo = scope.ServiceProvider.GetRequiredService<ITenantAwareRepository>();
    await repo.DoThing();
}, stopping);
```

If you have many pieces of ambient context, centralize the copy in a helper:

```csharp
public static class AmbientContextSnapshot
{
    public static Action<IServiceProvider> Capture(IServiceProvider parent)
    {
        var tenantId = parent.GetRequiredService<ITenantContext>().TenantId;
        var userId   = parent.GetRequiredService<IUserContext>().UserId;
        var correlationId = parent.GetRequiredService<ICorrelationContext>().CorrelationId;

        return child =>
        {
            child.GetRequiredService<ITenantContext>().TenantId = tenantId;
            child.GetRequiredService<IUserContext>().UserId = userId;
            child.GetRequiredService<ICorrelationContext>().CorrelationId = correlationId;
        };
    }
}

// Usage:
var applyContext = AmbientContextSnapshot.Capture(parentSp);

var task = Task.Run(async () =>
{
    await using var scope = scopeFactory.CreateAsyncScope();
    applyContext(scope.ServiceProvider);
    // ... rest of fire-and-forget work
}, stopping);
```

This helper is the functional replacement for `IEventScopeInitializer` — consumer-owned, explicit, and easy to audit.

---

### Authorization flags

Every `[AuthorizeFactory(AuthorizeFactoryOperation.Event)]` and `AuthorizeFactoryOperation.Event | Foo` reference must be deleted. Authorization on the factory methods that formerly triggered the event delegate (typically `[Create]` / `[Execute]`) still applies unchanged — the fire-and-forget work inside `Task.Run` is just server-side code and is already inside an authorized request scope.

### Remove DI calls

```csharp
// DELETE these calls if present — the methods / types no longer exist:
services.AddRemoteFactoryEventScopeInitializer<MyInitializer>();
// anything referencing IEventTracker / EventTrackerHostedService / IEventScopeInitializer
```

`AddNeatooRemoteFactory(...)` no longer registers `IEventTracker` or `IEventScopeInitializer`. `ICorrelationContext` is still registered scoped in all modes.

### Remove test references

Any test helpers that injected `IEventTracker` (typically a `WaitAllAsync` at the end of a test) now wait on your own `IFireAndForgetTracker` or, where the test doesn't spawn fire-and-forget work at all, can remove the wait entirely. The RemoteFactory integration-test helper `IEventTestService` was preserved and moved out of the deleted `EventTargets.cs` — it's still available for `[FactoryEventHandler<T>]` tests.
