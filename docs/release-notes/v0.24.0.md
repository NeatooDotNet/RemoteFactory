# v0.24.0 - Trimming-Safe DTO Serialization

**Release Date:** 2026-03-25
**NuGet:** [Neatoo.RemoteFactory 0.24.0](https://nuget.org/packages/Neatoo.RemoteFactory/0.24.0)

## Overview

The source generator now automatically preserves plain DTO return types from IL trimming. DTOs returned by factory methods (Interface, Class, or Static factories) survive trimming without any manual configuration — constructors, properties, and all metadata needed by `System.Text.Json` are preserved.

This supersedes v0.23.2's `Activator.CreateInstance` approach, which did not work under trimming.

## What's New

- **Generator-emitted DTO constructor registration.** The source generator discovers plain DTO classes returned by factory methods and emits `DtoConstructorRegistry.Register<T>(() => new T())` calls at compile time. The `[DynamicallyAccessedMembers(All)]` annotation ensures the IL trimmer preserves the entire type. This covers:
  - Interface Factory method return types (e.g., `Task<EmployeeDto>`, `Task<IReadOnlyList<EmployeeDto>>`)
  - Class Factory `[Execute]` return types
  - Static Factory `[Execute]` return types
  - Automatic unwrapping of `Task<T>`, nullable `T?`, and collection types

- **DtoConstructorRegistry.** A new static registry (`Neatoo.RemoteFactory.Internal.DtoConstructorRegistry`) that holds constructor lambdas for discovered DTO types. Populated during `AddNeatooRemoteFactory()` / `AddNeatooAspNetCore()` startup. Used by `NeatooJsonTypeInfoResolver` to set `CreateObject` without reflection.

- **Trimming documentation.** New "DTO Return Type Preservation" section in the [IL Trimming](../trimming.md) documentation explaining why `IsTrimmable=true` breaks `DefaultJsonTypeInfoResolver` for plain DTOs and how the generator handles it automatically.

## Bug Fixes

- **Fixed DTO deserialization under IL trimming** ([#48](https://github.com/NeatooDotNet/RemoteFactory/issues/48)). Plain DTO classes returned by factory methods (e.g., `ExampleDto` via an Interface Factory) now deserialize correctly in trimmed Blazor WASM applications. Previously failed with `DeserializeNoConstructor` because the trimmer stripped constructor and property metadata.

- **Replaced `Activator.CreateInstance` fallback** (v0.23.2). The previous fix used `Activator.CreateInstance` as a runtime fallback, which also fails under trimming. The generator-based approach creates static references the trimmer can trace.

- **Fixed Design.Client.Blazor script tag.** The `index.html` had a broken `blazor.webassembly#[.{fingerprint}].js` placeholder that was never replaced during publish.

## Why This Matters

Normal Blazor WASM apps don't hit this because their assemblies aren't trimmed — `TrimMode=partial` only trims assemblies marked `IsTrimmable=true`. RemoteFactory intentionally marks domain assemblies as trimmable to remove server-only business logic (factory method bodies, EF Core, repositories) from the client. This is a core value proposition: one assembly with trimming instead of separate client/server projects.

The trade-off is that `System.Text.Json`'s `DefaultJsonTypeInfoResolver` uses reflection to discover constructors and properties, which fails on trimmed types. The generator closes this gap by creating static references for DTO types that cross the client/server boundary.

## Workaround Removal

If you previously added a `TrimmerRootDescriptor` XML to preserve DTO types manually, you can remove it after upgrading:

```xml
<!-- No longer needed with v0.24.0 -->
<linker>
  <assembly fullname="MyApp.DomainModels">
    <type fullname="MyApp.DomainModels.MyDto" preserve="all" />
  </assembly>
</linker>
```
