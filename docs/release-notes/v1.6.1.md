# v1.6.1 — `[Execute]`-only Factory Registration Fix

**Release Date:** 2026-05-24
**NuGet:** [Neatoo.RemoteFactory 1.6.1](https://nuget.org/packages/Neatoo.RemoteFactory/1.6.1)
**Breaking changes:** None.

---

## Overview

Fixes a generator bug where `[Factory]` classes whose only operation is `[Execute]` and whose constructor requires DI services were not registered as transients in the generated `FactoryServiceRegistrar`. On the client, the JSON deserializer could not resolve the type via `IServiceProvider.GetRequiredService`, so the constructor never ran during deserialization and any constructor-injected service stayed `null` on the deserialized instance.

The aggregate shape that exposed this: an internal `[Factory]` class with init-only properties, exposed via a public interface, whose only operation is `[Remote, Execute] public static Task<TInterface> Open(...)`. Common for stateless "open / start / process" workflows whose entry point doesn't fit Create / Fetch semantics.

## Bug Fixes

### `[Execute]`-only entity not registered, constructor-injected services missing on client

`FactoryModelBuilder.cs` scoped its `requiresEntityRegistration` flag to `ReadMethodModel` only (Create / Fetch). A class that exposed only `[Execute]` operations never got `AddTransient<EntityType>()` in its `FactoryServiceRegistrar`, even when the constructor required DI to resolve.

```csharp
// Before — generator emitted only the factory registrations, not the entity itself.
// The client-side deserializer then could not invoke this ctor through DI.
[Factory]
internal partial class TreatmentContext : ITreatmentContext
{
    public TreatmentContext(ITreatmentFactory factory) { _factory = factory; }

    [Remote, Execute]
    public static Task<ITreatmentContext> Open(int id, [Service] IServiceProvider sp) { ... }
}
```

The flag now also fires when `typeInfo.RequiresServiceInstantiation` is `true` — i.e., when the type's constructor cannot be invoked with zero arguments. Generated `FactoryServiceRegistrar` for such an `[Execute]`-only entity now includes the missing pair:

```
services.AddTransient<TreatmentContext>();
services.AddTransient<ITreatmentContext, TreatmentContext>();
```

so `NeatooJsonTypeInfoResolver` can resolve the type during deserialization and the constructor runs on each side of the wire with the local container's services.

---

## Migration Guide

No action required. Rebuild your project; the generator will emit the missing `AddTransient` calls automatically. If you previously worked around the bug by manually registering the entity type, you can remove that workaround — the duplicate registration is harmless but no longer needed.
