# Explore IL Trimming Feature Switches for Remote-Only Code Separation

**Status:** In Progress
**Priority:** Medium
**Created:** 2026-03-02
**Last Updated:** 2026-03-02 (updated with reachability findings)

---

## Problem

Blazor WASM publish fails or produces bloated output when server-only code (EF Core, repositories, database access) is referenced from shared assemblies. The current workaround is splitting assemblies (e.g., `Person.Dal` separate from `Person.Ef`), which adds project complexity.

---

## Solution

Use .NET 9+ `[FeatureSwitchDefinition]` and `[FeatureGuard]` attributes to conditionally remove server-only code during IL trimming. The trimmer treats guarded properties as compile-time constants, removing entire code branches and their transitive dependencies from the published output.

### Key Pattern

```csharp
public static class NeatooRuntime
{
    [FeatureSwitchDefinition("Neatoo.IsServerRuntime")]
    public static bool IsServerRuntime =>
        AppContext.TryGetSwitch("Neatoo.IsServerRuntime", out bool v) ? v : true;
}

// In factory/data portal code:
public async Task<T> Fetch(...)
{
    if (NeatooRuntime.IsServerRuntime)
    {
        // Server-only: EF Core, database access, etc.
        // Completely removed from Blazor WASM publish
        return await repository.LoadFromDb(...);
    }
    else
    {
        return await remoteProxy.FetchFromServer(...);
    }
}
```

Blazor WASM csproj configuration:
```xml
<RuntimeHostConfigurationOption Include="Neatoo.IsServerRuntime" Value="false" Trim="true" />
```

### All Relevant Trimming Attributes

| Attribute | Purpose | Removes Code? |
|---|---|---|
| `[RequiresUnreferencedCode]` | Warns callers about trim incompatibility | No, just warns |
| `[DynamicallyAccessedMembers]` | Preserves specific reflection targets | No, preserves |
| `[DynamicDependency]` | Force-preserves specific symbols | No, preserves |
| `[UnconditionalSuppressMessage]` | Suppresses trim warnings in IL | No, just suppresses |
| `[FeatureSwitchDefinition]` | **Conditionally removes guarded code** | **Yes** |
| `[FeatureGuard]` | Guards `Requires*` code + enables removal | **Yes** |

### How It Works

1. `Trim="true"` on `RuntimeHostConfigurationOption` tells the trimmer to substitute `IsServerRuntime` with constant `false`
2. Trimmer evaluates `if (false)` and marks the entire block as unreachable
3. Unreachable code is removed from output
4. Trimmer follows the dependency graph â€” anything **only** reachable through removed branches also gets trimmed (EF types, repository classes, etc.)

### Member-Level Trimming: What Gets Removed?

With `TrimMode=full` (default for Blazor WASM since .NET 8), the trimmer works at **member level**, not type level. If a class is referenced but a specific method is never called from any reachable code path, that method **and its parameter type dependencies** get trimmed.

```
PersonEdit is referenced           âś“  (kept - used in UI)
PersonEdit.Name property           âś“  (kept - used in UI)
PersonEdit.Insert(AppDbContext)    âś—  (trimmed - nobody calls it)
  â””â”€ AppDbContext dependency       âś—  (also trimmed if nothing else references it)
    â””â”€ EF Core types               âś—  (also trimmed transitively)
```

**Key insight:** Just because a class is used doesn't mean all its methods survive. Only **reachable** members are kept.

### Critical Caveat: What Makes Methods "Reachable"

A method stays alive if ANY of these are true:

1. **Called directly** from reachable code
2. **Virtual/override where the base virtual is called polymorphically** â€” if anyone calls `base.DoSomething()` through a `EntityBase<T>` reference, the trimmer conservatively keeps ALL overrides across ALL derived types
3. **Called through reflection** â€” `Type.GetMethod("Insert")` etc.
4. **Called by generated code** â€” RemoteFactory's output references the method

#### The RemoteFactory Problem

This is the critical point for Neatoo. If RemoteFactory generates:

```csharp
// Generated factory method (current behavior)
public async Task<PersonEdit> Fetch(int id)
{
    var obj = new PersonEdit();
    await obj.DataPortal_Fetch(id, dbContext);  // <-- Makes DataPortal_Fetch REACHABLE
    return obj;
}
```

Then `DataPortal_Fetch(int, AppDbContext)` is reachable even in WASM, because the generated factory code references it. The trimmer keeps it, and AppDbContext/EF stay in the output.

#### Where Feature Switch Guards Must Go

The guards need to be in **RemoteFactory's generated code**, not just in application code:

```csharp
// Generated by RemoteFactory (proposed change)
public async Task<PersonEdit> Fetch(int id)
{
    if (NeatooRuntime.IsServerRuntime)
    {
        // Server path - calls the [Fetch] method with DbContext
        var obj = new PersonEdit();
        await obj.DataPortal_Fetch(id, dbContext);  // TRIMMED on WASM
        return obj;
    }
    else
    {
        // Client path - calls remote server
        return await remoteProxy.Fetch<PersonEdit>(id);  // KEPT on WASM
    }
}
```

With `IsServerRuntime = false` in the WASM csproj, the trimmer:
1. Removes the dead `if (false)` branch entirely
2. `DataPortal_Fetch(int, AppDbContext)` becomes unreachable
3. `AppDbContext` becomes unreachable (if nothing else references it)
4. EF Core types pulled in by `AppDbContext` become unreachable

#### The Virtual Method Risk

If `EntityBase<T>` declares server-only operations as `virtual`:

```csharp
public class EntityBase<T> {
    internal virtual void Insert(AppDbContext db) { }  // BAD for trimming
}
```

The trimmer must be conservative â€” if `Insert` is called on *any* `EntityBase<T>` reference, it preserves ALL overrides. This defeats trimming.

**However** â€” Neatoo's `[Insert]`/`[Update]`/`[Delete]` methods are defined on concrete application classes, not as overrides of base class virtuals. If RemoteFactory calls them on the concrete type (not through a base reference), the trimmer can reason about them individually.

### Where to Apply Guards: Impact Summary

| Where guards are placed | Effect |
|---|---|
| Application code calling factories | Partial â€” only trims your call sites |
| **RemoteFactory generated code** | **Full â€” trims server methods + their dependencies** |
| EntityBase internal methods | Risky if virtual/interface-based |

**Conclusion:** The most powerful approach is having **RemoteFactory's source generator emit feature-switch-guarded code**. The generator already knows which methods are `[Remote]` vs local â€” it could emit the `if (IsServerRuntime)` guards automatically. This would eliminate the need for assembly splits entirely; the trimmer does the separation.

### `[FeatureGuard]` for Suppressing Warnings

```csharp
public static class NeatooRuntime
{
    [FeatureGuard(typeof(RequiresDynamicCodeAttribute))]
    internal static bool IsDynamicCodeSupported =>
        RuntimeFeature.IsDynamicCodeSupported;
}

// No analyzer warnings when called inside the guard:
if (NeatooRuntime.IsDynamicCodeSupported)
    DoDynamicStuff(); // Trimmed away in AOT/WASM scenarios
```
